Ось стисло, чого не вистачає у поточному шаблоні, щоб говорити про event-driven orchestration (оркестрація/хореографія подій, довгі процеси) як про реалізовану здатність — не як про майбутній намір з ADR 0008.

1. Інфраструктура повідомлень
   Брокер (Kafka, RabbitMQ, NATS JetStream, Pulsar тощо) у local dev / CI / compose — зараз немає.
   Топіки / черги, політики retry + DLQ, схеми повідомлень (Avro/JSON Schema, реєстр схем).
   Контракт між продюсерами й консьюмерами (async API так само важливий, як OpenAPI для HTTP).
2. Патерни в коді застосунку
   Outbox (транзакція БД + запис події в таблицю → фоновий пабліш) — щоб не губити події між комітом і брокером.
   Ідемпотентність консьюмерів (ключі, дедуплікація, “exactly-once” як ціль через idempotent handlers).
   Явна модель saga / компенсаційні дії (оркестратор у сервісі або workflow engine ззовні — див. нижче).
   Розділення команд / подій / запитів (часто CQRS-light для read-моделей).
3. Оркестрація vs двигун workflow
   Потрібно вибрати рівень:

Легка хореографія — лише події між сервісами (без центрального “диригента”) — все одно потрібні чіткі контракти, версіювання подій, спостережуваність.
Durable workflows / orchestration — зазвичай окремий Temporal / Camunda / Conductor / Durable Task (або керований SaaS), плюс воркери, версіювання workflow, replay, таймаути й компенсації.
У шаблоні немає ні брокера, ні workflow-движка — тобто немає носія для “оркестрації” в сенсі довгоживучих процесів.

4. Спостережуваність для асинхрону
   Трейсинг через черги (propagation trace context у message headers).
   Метрики: lag, consumer lag, replay rate, DLQ depth.
   Логи: correlation id на рівні події (не лише HTTP requestId).
   Зараз стек заточений під синхронний запит; для подій потрібні додаткові панелі/алерти.

5. Тестування та CI
   Testcontainers для брокера в інтеграційних тестах.
   Контрактні тести async (Pact message contracts, або schema registry checks).
   Chaos / load на консьюмерів (повільні партиції, ребаланс).
6. Операційні практики
   Runbook для “застряглих” подій, replay, poison messages.
   Політика версіювання подій (backward-compatible зміни, dual-write періоди).
   Підсумок: для event-driven orchestration не вистачає головним чином (а) транспорту подій і його експлуатації, (б) патернів у коді (outbox, idempotency, saga/workflow-модель), (в) спостережуваності та тестів під асинхронний шлях. Це узгоджується з ADR 0008 (фази з messaging, idempotency, outbox) — це якраз той беклог, який треба закрити, щоб “йти до event-driven” не на словах, а в репозиторії.
