# План налаштування Grafana (і суміжних речей)

**Призначення:** зібрати в одному місці **що ще можна зробити** у стеку спостереження навколо Grafana і **які рішення вже прийняті / що лишається відкритим**. Доповнює [monitoring-optimization-plan.md](monitoring-optimization-plan.md) (технічний чеклист).

**Статус репозиторію зараз:** datasource-и (Prometheus, Loki, Jaeger), один provisioned дашборд `quarkus-overview`, Jaeger → Loki / Prometheus (`tracesToLogsV2`, `tracesToMetrics`), Alloy → Loki. Деталі — [ADR 0003](../adr/0003-observability-monitoring-stack.md).

### Впроваджено (фази A–C)

| Що | Де |
|----|-----|
| Явний `job="quarkus-ms-gold-template"` у всіх PromQL-запитах на дашборді (узгоджено з `deploy/monitoring/prometheus.yml` і `jaeger.yaml`) | `grafana/dashboards/quarkus-overview.json` (version 2) |
| Derived field `traceId` → Jaeger у Loki datasource | `grafana/provisioning/datasources/loki.yaml` (вже було) |
| Чеклист перевірки A–C + типові проблеми | [grafana-verify.md](grafana-verify.md) |
| Швидка перевірка здоров’я стеку | `make verify-observability` |
| Узгодження datasource-ів з документацією Grafana 11.6 | [grafana-prometheus-review.md](grafana-prometheus-review.md) |

Далі за планом: **D** (алерти + канал), **E** (recording rules / exemplars), **F** (прод-безпека) — за потреби.

---

## 1. Мета плану

| Ціль | Як зрозуміти успіх |
|------|-------------------|
| **Комфортна розробка** | Локально можна швидко підняти стек і подивитися метрики, логи, трейси без ручної збірки |
| **Дослідження проблем** | Є доступ до **усіх потрібних сигналів** (метрики, логи, трейси, зв’язки між ними) і зрозумілі запити в Explore |
| **Dev-сервер** | Окреме середовище для **відстеження помилок і тестування** спостережуваності без навантаження продакшену |
| **Шаблон** | Конфіг у git (provisioning), щоб відтворити однакову картину на іншій машині |

Алерти та зовнішні канали сповіщень — **пізніший крок** (див. §2.5).

## 2. Рішення та відкриті теми

### 2.1 Середовище та аудиторія — **рішення для цього шаблону**

| Контекст | Рішення |
|----------|---------|
| **Локальна розробка** | Grafana + стек з `deploy/docker-compose-metrics.yml` на машині розробника: перегляд метрик, логів (через Loki/Alloy для контейнерів), трейсів у Jaeger з Grafana |
| **Dev-сервер** | Окремий хост/VM з тим самим підходом (compose або K8s) для **відтворення помилок**, перевірки дашбордів і інтеграцій **до** продакшену; не змішувати з бойовим навантаженням |
| **Аудиторія** | На старті — **команда розробки** (повний доступ для дослідження). Ширший доступ (viewer-only) — коли з’явиться потреба |
| **Один Grafana на багато сервісів** | У шаблоні закладено **один** інстанс і папки/дашборди за потреби; окремий Grafana на кожен мікросервіс — не вимога для цього репозиторію |

**Висновок:** «Прод Grafana з SSO» виходить за рамки поточного шаблону; фокус — **локально + dev для дебагу та тестів**.

---

### 2.2 Метрики та Prometheus — **в чому тут питання (простими словами)**

Тут не «філософські» питання, а **технічна узгодженість**, щоб у Grafana не було порожніх панелей і щоб **переходи з Jaeger у метрики** працювали.

| Що саме може зламатися | Що перевірити / тримати узгодженим |
|------------------------|-------------------------------------|
| **Ім’я job у Prometheus** | У `deploy/monitoring/prometheus.yml` для scrape задано `job` (наприклад `quarkus-ms-gold-template`). У **дашбордах**, у **Jaeger `tracesToMetrics`**, у змінних Grafana має бути **той самий** рядок — інакше PromQL повертає порожньо або «No data». |
| **Змінні дашборду** | Змінна `instance` / інші будуються з метрик, які **реально існують** після старту застосунку (`up`, `process_cpu_usage`, …), а не з рідкісних серій. |
| **SLO (доступність, p95, помилки)** | Це **не** обов’язкова налаштування Grafana зараз; це **орієнтири**, коли додаватимете алерти. Референс — [ADR 0002](../adr/0002-sre-production-readiness.md). |
| **Recording rules** | Опційно: якщо якісь панелі з важким PromQL почнуть гальмувати при великих діапазонах часу — винести агрегацію в Prometheus `rules.yml`. |
| **Exemplars** | Опційно: точка на графіку метрики → перехід у трейс; потрібна підтримка в експорті метрик + налаштування Prometheus datasource у Grafana. |

**Підсумок:** головне питання §2.2 — **однакові лейблі та імена в scrape, дашбордах і `tracesToMetrics`**. Решта — коли з’явиться навантаження або алерти.

### 2.3 Логи та Loki — **що варто знати (дослідження)**

| Тема | Суть для цього проєкту |
|------|-------------------------|
| **Формат** | У prod увімкнено JSON у консоль (`%prod.quarkus.log.console.json.enabled`) — зручно парсити в Loki. Перевірити в Explore один реальний рядок: чи є поля `traceId` / `requestId` так, як очікує [logging-loki-alloy.md](logging-loki-alloy.md). |
| **Як лог потрапляє в Loki** | Локально з `make up-prod`: Alloy зчитує **Docker-логи** контейнера бекенду → Loki. `quarkus:dev` **на хості** у цей пайплайн за замовчуванням **не** потрапляє — це обмеження, не баг; для дослідження логів зручніше дивитися файл або тимчасово підняти бекенд у контейнері. |
| **Derived fields у Grafana** | Якщо в логах стабільне поле `traceId`, у datasource Loki можна налаштувати **перехід у Jaeger** одним кліком (regex/JSON під ваш формат). Варто перевірити після першого деплою на dev-сервер. |
| **Retention** | На локалці та типовому dev **жорсткі квоти Loki не критичні**; для довго живучого dev-серверу має сенс обмежити retention у `deploy/monitoring/loki-config.yaml`, щоб диск не ріс безконтрольно. |

---

### 2.4 Трейси та кореляція — **що варто знати (дослідження)**

| Тема | Суть |
|------|------|
| **Вже є в репо** | У Jaeger datasource налаштовано **tracesToLogsV2** (трейс → логи в Loki) і **tracesToMetrics** (трейс → PromQL у Prometheus). Цього зазвичай достатньо для дебагу. |
| **Grafana Correlations** | Додатковий шар: узгоджені переходи між **будь-якими** джерелами за правилами. Має сенс, якщо UX «стрибків» між Explore стане тісним; для шаблону — **не обов’язково** спочатку. |
| **Семплінг** | У `%prod` трейси семплюються (наприклад 25%) — частина запитів у Jaeger **без** повного трейсу. Для dev часто лишають `always_on`, щоб не губити трейси під час розслідування. |

### 2.5 Алерти — **рішення для шаблону**

| Рішення |
|---------|
| **Канал сповіщень** (Slack, email, PagerDuty тощо) **не фіксуємо зараз** — налаштуємо пізніше під реальний on-call. |
| **Базовий шаблон:** достатньо узгоджених дашбордів і Explore; опційно — заготовка правил алертів у Grafana **без** contact point або з тестовим webhook, коли з’явиться §2.2 SLO. |

---

### 2.6 Безпека — **навіщо це взагалі для Grafana**

Питання не про «чи любимо ми безпеку», а про **що саме захищаємо**:

| Що | Навіщо |
|----|--------|
| **Пароль admin у compose** | Grafana зберігає налаштування, іноді datasource з URL; **відкритий порт 3000 у мережі** з дефолтним `admin/admin` — це ризик. На **localhost** для одного розробника зазвичай прийнятно; на **dev-сервері в інтернеті** — змінити пароль, обмежити доступ (firewall, VPN). |
| **HTTPS / домен** | Потрібні, коли Grafana **не лише на 127.0.0.1**: щоб паролі й сесії не йшли відкритим текстом. Для чисто локального `localhost` — не обов’язково. |

Тобто §2.6 — **коли виносите Grafana з ноутбука в мережу**, а не вимога для комфортної локальної розробки.

---

### 2.7 Ресурси та бекапи — **пріоритет для цього шаблону**

| Контекст | Позиція |
|----------|---------|
| **Локальна розробка** | Жорсткі **CPU/RAM limits** для Prometheus/Loki/Grafana і **бекапи** томів — **не супер-критичні**; достатньо не тримати без потреби місячні дані на ноутбуку. |
| **Спільний dev-сервер** | Має сенслегка обмежити контейнери і подумати про **retention** Loki / Prometheus, щоб диск не заповнився. |

## 3. Фази A–F: що означають і best practice

**Ідея:** фази — це **логічний порядок залежностей**, а не жорсткий регламент. Спочатку має бути **видимість даних**, потім **зручність дебагу**, потім **алерти**, потім **посилена безпека для прод**.

| Фаза | Назва | Що робити | Навіщо саме так |
|------|--------|-----------|-----------------|
| **A** | **Дані видно** | Підняти стек (`make up-metrics` + застосунок), переконатися: Prometheus scrape **up**, у Grafana Explore є метрики / логи / трейси | Без цього решта налаштувань не перевірити |
| **B** | **Узгоджені імена** | Перевірити `job`, запити на дашборді, `tracesToMetrics` у `jaeger.yaml` — один набір лейблів | Уникнути «No data» і зламаних переходів (див. §2.2) |
| **C** | **Кореляція в дії** | Пройти сценарій: помилка → лог у Loki → traceId → Jaeger → назад метрики; за потреби derived fields у Loki | Переконатися, що **пошук проблем** end-to-end реальний |
| **D** | **Алерти (коли буде потреба)** | Визначити SLO, правила в Grafana, потім contact points | Алерти без порогів і каналу лише шумлять — тому після A–C і після вашого рішення з §2.5 |
| **E** | **Оптимізація** | Recording rules, exemplars, анотації деплоїв на графіках | Коли з’являться важкі запити або потреба в глибшій діагностиці |
| **F** | **Прод-хардненінг** | SSO, HTTPS, секрети, бекапи, HA | Коли Grafana стає **критичним** корпоративним інструментом, не dev-інстансом |

**Best practice коротко:** не налаштовувати алерти (D), поки не працює стабільний шлях A→B→C; не вимагати HTTPS (F) для сценарію «лише localhost».

## 4. Критерій готовності для шаблону репозиторію

Шаблон вважається **достатнім для цілей цього проєкту**, якщо:

- [ ] Розробник **комфортно** піднімає локально стек спостереження і застосунок і бачить **узгоджену** картину (метрики + за потреби логи/трейси).
- [ ] Можна **дослідити різні сервіси** стеку (Prometheus, Grafana, Loki, Jaeger, Alloy) через документацію репо та Explore — без «чорних скриньок».
- [ ] Є **усі необхідні параметри для пошуку типових проблем**: RPS/латентність/помилки на метриках, логи з кореляцією до трейсу де налаштовано, трейси в Jaeger з посиланнями на логи/метрики.
- [ ] Дашборди та `metrics-inventory.md` **не суперечать** реальним іменам з `/q/metrics`.
- [ ] Канал алертів і прод-безпека — **за потреби пізніше** (узгоджено з §2.5 і §2.6).
